# agent/ — харнесс базы знаний ReseachOS

Канонический каталог. Симлинк `kb/` → `agent/` сохранён для старых путей
(`python agent/bin/build_kb.py`, ссылки в скиллах).

Назначение: стандартизировать проверку гипотез и накапливать проверенные знания так,
чтобы новый человек или агент за минуты получал контекст «что мы уже знаем и как тут
работать», а каждый новый эксперимент пополнял этот контекст.

## Главный принцип: один источник правды

KOI-проекты — источник правды. `agent/` ничего не дублирует, а добавляет недостающие слои:

**Старт агента:** [AGENTS.md](AGENTS.md). **Документация:** `docs/human/` (люди), `docs/agent/` (техсправка).

| Слой | Где живёт | Кто пишет |
|------|-----------|-----------|
| Гипотезы, канбан, вердикты | `projects/<id>/project.md` | человек/агент (UI или md) |
| Инсайты (≤3 на метод) | `projects/<id>/research.json` | человек/агент (UI или md) |
| Публичные отчёты | `projects/<id>/reports/` по `standards/templates/report-skeleton.md` | человек/агент |
| **Пред-регистрация гипотезы** | `agent/templates/hypothesis-spec.md` → описание cause-узла | автор гипотезы |
| **Рабочий отчёт о прогоне** | `reports/<узел>/<карточка>.run.md` по `agent/templates/experiment-report.md` | агент-исполнитель |
| **База знаний проекта: документы** | `projects/<id>/knowledge/*.md` (обзор, установка, запуск, скрипты, грабли…) | человек/агент; `hypotheses.md` — автоген |
| **База знаний проекта: оглавление** | `projects/<id>/KNOWLEDGE.md` — сводки + ссылки, не раздувается | **генерируется** `koi/knowledge.py` (хук `save_project`) |
| **База знаний проекта: журнал** | `projects/<id>/KNOWLEDGE_LOG.md` — что и когда записалось | **генерируется** (автодифф при сохранении) |
| **Глобальный индекс знаний** | `agent/KNOWLEDGE.md` | **генерируется** `agent/bin/build_kb.py` |
| **Онбординг** | `agent/onboarding/` | repo-map, commands (продукт) |
| **Процесс + матрица решения** | `agent/process.md` | — |

База знаний КАЖДОГО проекта устроена так, чтобы один файл не раздувался:

```
projects/<id>/
  KNOWLEDGE.md       оглавление: статистика, сводка каждого документа + ссылка,
                     статус гипотез (генерируется — не править)
  KNOWLEDGE_LOG.md   журнал пополнений: что и когда записалось (генерируется)
  knowledge/         полные документы знаний (.md):
    hypotheses.md    АВТОГЕН: гипотезы + вердикты + инсайты + ссылки на отчёты
    <NN-имя>.md      курируемые: обзор, установка, запуск, скрипты, грабли,
                     открытые вопросы… (конвенция: `# Заголовок`, первый абзац = сводка)
```

**Автоматическая интеграция новых знаний** — три пути, все без ручной сборки:
1) агент меняет вердикт в `project.md` / добавляет инсайт в `research.json` →
   `save_project` пересобирает `hypotheses.md` + оглавление и дописывает журнал;
2) новый `.md` кладётся в `knowledge/` → при следующем сохранении (или
   `python agent/bin/build_kb.py`) он появляется в оглавлении и в журнале;
3) **отчёт `.run.md` вливается автоматически**: `koi/report_ingest.py` парсит §5
   (вердикт + json-блок инсайтов) и сам обновляет вердикт узла, `research.json`,
   карточку (→ done) и `reports/index.json` — дальше срабатывает путь (1).

**Автоматическая проверка гипотезы агентом** — одна команда:
`PYTHONPATH=. python scripts/koi_check_hypothesis.py <project_id> <card_id>` —
агент (Claude Code CLI или Cursor SDK, слой `koi/agent_backends.py`) пишет
рабочий отчёт по `agent/templates/experiment-report.md` и отчёт тут же вливается в БЗ.
Подробно, включая переменные окружения бэкендов, — `docs/human/knowledge-base.md`.

Глобальный `agent/KNOWLEDGE.md` — тонкий индекс-ссылочник по проектам. Доступ: файлы в
репо; API `GET /projects/{id}/knowledge` (оглавление), `…/knowledge/log` (журнал),
`…/knowledge/file?path=…` (любой документ); в веб-интерфейсе — кнопка «База знаний».

## Цикл знаний

```
1. Вопрос → agent/templates/hypothesis-spec.md (claim, предсказание, ПРАВИЛО РЕШЕНИЯ, бюджет)
2. Спека → узел cause в project.md + карточка канбана (backlog)
3. Прогон: карточка running → done
4. Рабочий отчёт о прогоне по agent/templates/experiment-report.md
   (reports/<узел>/<карточка>.run.md) с блоком «Заявка в базу знаний»
5. Ревью заявки по process.md → решение, В КАКОМ ВИДЕ интегрировать (матрица решения)
6. Публичный отчёт по report-skeleton.md + инсайты в research.json + вердикт узла
7. Авто: save_project → knowledge/hypotheses.md + KNOWLEDGE.md (оглавление) +
   запись в KNOWLEDGE_LOG.md; build_kb.py → глобальный индекс
8. Следующая гипотеза начинается с чтения KNOWLEDGE.md проекта (и knowledge/
   50-open-questions.md, если есть) → знание компаундится, опыт не открывают заново
```

## Файлы

- `AGENTS.md` — главная инструкция агента IDE (очереди, циклы, форматные ворота).
- `templates/hypothesis-spec.md` — пред-регистрация гипотезы (заполнить до запуска).
- `templates/experiment-report.md` — рабочий отчёт агента о прогоне + заявка в базу знаний.
- `templates/report.md` — как применять штатный `standards/templates/report-skeleton.md`.
- `process.md` — процесс «прогон → рабочий отчёт → ревью → вердикт → реестр», матрица решения, чеклист.
- `onboarding/` — repo-map, commands (продукт KOI).
- `bin/build_kb.py` — пересборка per-project `projects/<id>/KNOWLEDGE.md` + глобального `agent/KNOWLEDGE.md`.
- `KNOWLEDGE.md` — глобальный индекс баз знаний (генерируется, не править).

База знаний каждого проекта живёт в самом проекте: `projects/<id>/KNOWLEDGE.md`
(рендерит модуль ядра `koi/knowledge.py`).

## Контекст для новичка/агента

Минимальный брифинг по проекту = его `KNOWLEDGE.md` (оглавление: что знаем и где
лежит) → оттуда по ссылкам: установка, запуск, скрипты, грабли, гипотезы с
результатами, открытые вопросы. Общерепозиторное (не про конкретный проект) —
`agent/onboarding/` (продукт). Примеры и harness — `examples/`. Несущий RL-пример —
проект `isaac-rl-bench` в workspace и `examples/isaac_harness/`.
