# ResearchOS с нуля: от идеи до базы знаний

Сквозной путеводитель на одном учебном примере — «Решение квадратных уравнений».
Проходит весь цикл: создать проект в веб-интерфейсе → расписать дерево гипотез →
завести эксперименты в канбане → провести эксперимент → написать отчёт →
автоматически влить результат в базу знаний → посмотреть, что получилось.

Каноничные справочники (этот документ — маршрут по ним, а не замена):

| Что | Где |
|-----|-----|
| Формат проекта и дерева | `docs/human/project-format.md` |
| Формат программ | `docs/human/program-format.md` |
| Устройство базы знаний целиком | `docs/human/knowledge-base.md` |
| Шаблон рабочего отчёта (главный!) | `agents/skills/koi-report-review/experiment-report.md` |
| Шаблон публичного отчёта | `agents/skills/koi-report-review/report-skeleton.md` |

| Инструкции для агента IDE | `AGENTS.md` |

## 0. Самый быстрый старт — создай учебный проект

Для первого прохода создай небольшой учебный проект, в котором эксперимент можно
выполнить одной локальной командой. Например, сравни два способа решения квадратных
уравнений по времени и точности:

> **📘 Пример: решение квадратных уравнений** (программа «📘 Пример: квадратные уравнения»).
> Это образец, а не рабочий проект — **открывай узлы и карточки и тыкай что угодно,
> сломать ничего нельзя.**

Что в нём посмотреть (по одному клику):

1. **Карта гипотез.** Две ветки от проблемы:
   - «Перебор по сетке фундаментально неэффективен» — с зелёным бейджем ✔
     (гипотеза **подтверждена**);
   - «Наивный дискриминант теряет точность при большом b» — **без бейджа**
     (гипотеза **открыта**, ещё не проверена). Так выглядит незакрытая ветка.
2. **Готовый отчёт.** У метода «Решатель по Виета против перебора» в канбане есть
   карточка в колонке **done** — кликни по ней и увидь **полностью написанный отчёт**
   (что запускали, числа, вывод). Вот так выглядит правильно заполненный отчёт.
3. **Пустая карточка = готовый шаблон.** Кликни по карточке в **backlog** того же
   метода — редактор откроется с **преднаполненным шаблоном**: id и дата уже
   подставлены, осталось вписать результаты. Вот с чего ты начнёшь свой эксперимент.
4. **База знаний.** Кнопка **«База знаний»** в тулбаре → выбери этот проект: увидишь
   вердикт по гипотезе, два инсайта с числами и ссылку на отчёт-первоисточник.

Разобравшись на эталоне, заводи свой проект по шагам ниже.

## 1. Главная идея за 30 секунд

ResearchOS — это «Agile для науки». Вместо «решение → задачи» цикл такой:

```
ПРОБЛЕМА → причины (гипотезы о природе) → гипотезы (как доказать/устранить)
   → МЕТОДЫ проверки → карточки-эксперименты в канбане
   → ОТЧЁТ по эксперименту → вердикт + инсайты
   → БАЗА ЗНАНИЙ (собирается автоматически, без LLM)
```

Ключевое: **знание рождается только из эксперимента с заранее заданным правилом
решения**. Сначала пишешь «гипотеза подтверждена, если метрика X пройдёт порог Y»,
потом запускаешь, потом вердикт следует из подстановки чисел в правило — не из
общего впечатления.

Всё хранится в Markdown-файлах, без базы данных. Один проект = папка
`projects/<id>/`. Веб-интерфейс — просто удобный редактор этих файлов.

## 2. Запуск и вход

Стандартный способ (личная машина): `scripts/koi-serve.sh start` — поднимет API на
8010 и UI на 8080, дальше открыть http://localhost:8080.

**На airi-5090** порты 8010/8080 заняты чужим инстансом, поэтому свой поднимаем вручную:

```bash
# на сервере, из корня репо
source .venv/bin/activate
nohup .venv/bin/uvicorn api.main:app --host 127.0.0.1 --port 8011 > ~/.cockpit-jobs/koi-api-8011.log 2>&1 < /dev/null &
( cd web && nohup python3 -m http.server 8081 --bind 127.0.0.1 > ~/.cockpit-jobs/koi-web-8081.log 2>&1 < /dev/null & )
curl -sf http://127.0.0.1:8011/health    # → {"status":"ok",...}
```

С локального компа — туннель (UI ходит на API строго через локальный порт 8010):

```bash
ssh -N -L 8090:127.0.0.1:8081 -L 8010:127.0.0.1:8011 airi-5090
# UI: http://localhost:8090    API-сваггер: http://localhost:8010/docs
```

## 3. Словарь: из чего всё состоит

| Понятие | Что это | Где живёт на диске |
|---------|---------|--------------------|
| **Программа** | Стратегическое направление, объединяет проекты (группы в сайдбаре) | `programs/<id>/program.md` |
| **Проект** | Одна исследуемая проблема = одно дерево на карте | `projects/<id>/project.md` |
| **Проблема** (problem) | Корень дерева — что болит | заголовок `# problem:` в project.md |
| **Причина** (cause) | Гипотеза о *природе* проблемы. **Вердикты ставятся сюда** | `## cause: n-…` |
| **Гипотеза** (cause_evidence / remediation) | Как *доказать* причину / как её *устранить* | `### remediation: n-…` |
| **Метод** (method) | Конкретный способ проверки; у него канбан | `#### method: n-…` |
| **Карточка канбана** | Один эксперимент (backlog → running → done) | таблица под методом |
| **Отчёт** | Подробности эксперимента | `projects/<id>/reports/<узел>/<карточка>.md` |
| **Инсайт** | Вопрос-ответ с числами, итог эксперимента (≤3 на метод) | `projects/<id>/research.json` |
| **База знаний проекта** | Автособираемое оглавление: вердикты, инсайты, документы | `projects/<id>/KNOWLEDGE.md` + `knowledge/` |

Иерархия дерева жёсткая: problem → cause → (cause_evidence | remediation) → method.
UI сам не даст создать не туда — под каждым узлом пунктирный кружок «Добавить»
предлагает только допустимый тип.

## 4. Шаг 1 — создаём программу и проект (в UI)

1. Открой UI. Слева у края экрана — шеврон `<` / `>`: раскрывает **сайдбар программ
   и проектов**.
2. Внизу сайдбара — кнопка **«+ Новая программа»**. Назови её, например,
   `Решение квадратных уравнений`. Появится группа в сайдбаре, а на диске —
   `programs/решение-квадратных-уравнений/program.md`.
3. Рядом с заголовком программы — кнопка **«+»**: создаёт **новый проект внутри
   программы**. Имя проекта = формулировка проблемы, например
   `Вычислительная сложность перебора решений`.
   (Проект без программы попадает в группу «Без программы».)
4. Проект открылся на карте: один овальный узел «Проблема». На диске появилась папка
   `projects/вычислительная-сложность-перебора-решений/` с `project.md`,
   `research.json` и пустой базой знаний.

> id проекта/узлов генерируются автоматически (вид `n-1a2b3c4d`, `c-1a2b3c4d`, …).
> Они понадобятся для отчётов — где их смотреть, см. §7.2. Конкретные id ниже
> по тексту (`n-ce100001` и т.п.) взяты из **эталонного примера** (§0) — открой его
> и сверяйся.

## 5. Шаг 2 — расписываем дерево (что писать в узлах)

Под каждым узлом на карте есть пунктирный кружок **«Добавить +»** — он создаёт
ребёнка нужного типа («Добавить причину» → «Добавить доказательство или гипотезу» →
«Добавить метод»). Клик по узлу открывает панель: **«Редактировать»** — правишь
заголовок и описание, **«Сохранить»** / **«Отмена»**, внизу — **«Удалить узел»**.

Первая строка — короткий заголовок, дальше описание. Главное правило: **узел должен
быть проверяемым утверждением, а не темой**. «Аналитика» — плохо; «аналитическая
формула решает точнее и в ≥100 раз быстрее перебора» — хорошо.

Пример заполнения для нашего проекта — прямо текстом, как в жизни:

**Проблема** (корень, создаётся вместе с проектом):

> Вычислительная сложность перебора решений
>
> Решение ax²+bx+c=0 численным перебором кандидатов требует огромного числа
> проверок и всё равно не гарантирует точности. Хотим понять, почему перебор
> принципиально плох и чем его заменить.

**Причина** (кнопка «Добавить причину» под проблемой) — гипотеза о природе проблемы,
сформулированная так, чтобы её можно было подтвердить или опровергнуть:

> Бесконечное число вариантов для перебора
>
> Корни лежат в непрерывном пространстве: любая конечная сетка кандидатов либо
> промахивается мимо корня, либо требует астрономического числа проверок.
> Подтверждена, если на случайных уравнениях перебор с разумной сеткой проигрывает
> аналитике и по скорости (≥×100), и по точности (порядки residual).

**Гипотезы** (под причиной, «Добавить доказательство или гипотезу») — как устранить:

> Аналитическое решение
>
> Замкнутые формулы (дискриминант, теорема Виета) дают оба корня за O(1) с
> машинной точностью — перебор не нужен вовсе.

> Продвинутый солвер
>
> Готовые численные солверы (numpy.roots — собственные значения companion-матрицы)
> решают задачу без ручного перебора и масштабируются на старшие степени.

**Метод** (под гипотезой, «Добавить метод») — конкретный способ проверки, у него
появится канбан:

> Теорема Виета
>
> Реализовать решатель: x1+x2=−b/a, x1·x2=c/a (+ дискриминант для самих корней),
> сравнить с перебором по сетке на батче случайных уравнений: время на 1000
> уравнений и residual |f(x)|.

На карте у причин/гипотез появятся бейджи вердиктов (✔/✗), когда появятся первые
результаты, а у метода — строка канбана `tot · proc · done` с прогресс-баром.

## 6. Шаг 3 — заводим эксперименты (канбан у метода)

Клик по узлу-методу → в панели раздел «канбан» с колонками **backlog / running /
done**. В шапке каждой колонки кнопка **«+»** («Добавить карточку»). Карточки
перетаскиваются между колонками мышью — колонка и есть статус эксперимента.

Карточка = один эксперимент с понятным результатом. Для нашего метода:

| Колонка | Карточка |
|---------|----------|
| backlog | `Сравнить решатель по Виету с перебором по сетке на 1000 случайных уравнений` |
| backlog | `Проверить численную устойчивость при a≈0 и больших b (катастрофическое сокращение)` |

Правило хорошей карточки: из заголовка ясно, что будет сделано и что измерено.
«Проверка теоремы Виета» — слабо; «сравнить с перебором на 1000 уравнений: время и
residual» — то, что нужно.

Когда берёшь эксперимент в работу — перетащи карточку в `running`. В `done` её
руками таскать не обязательно: автоинтеграция отчёта (§8) перенесёт сама.

## 7. Шаг 4 — проводим эксперимент и пишем отчёт

Это сердце системы. Результат эксперимента попадает в базу знаний **только через
отчёт** строго определённого вида — рабочий `.run.md` по шаблону
`agents/skills/koi-report-review/experiment-report.md`. Заполнить его можно двумя путями.

### 7.1 Путь А — эксперимент проводит агент (одна команда)

```bash
# на сервере, из корня репо, в .venv
python -m koi.projects.report_ingest.cli <project_id> <card_id>
```

Скрипт сам соберёт контекст проекта (цепочку проблема→причина→гипотеза→метод),
выдаст агенту ТЗ с шаблоном отчёта, дождётся `.run.md` и автоматически интегрирует
его (вердикт, инсайты, карточка → done, база знаний). Полезные флаги:

| Флаг | Что делает |
|------|------------|
| `--backend claude` / `--backend cursor` | какой агент использовать (по умолчанию порядок из `KOI_AGENT_BACKEND`, дефолт `claude,cursor`) |
| `--no-ingest` | только сгенерировать отчёт, без интеграции (посмотреть глазами) |
| `--dry-run` | показать, что изменит интеграция, не трогая файлы |
| `--timeout 1800` | лимит работы агента в секундах |

Что нужно для бэкендов: Claude Code CLI (`claude login` или `ANTHROPIC_API_KEY`)
либо ключ Cursor API — его удобно положить через UI: кнопка **«Настройки»** в
тулбаре → «ключ Cursor API» (ляжет в `KOI/.env`, в git не попадает). Проверить, кто
доступен: `GET /agent/backends`.

Если серверного бэкенда нет (обычный случай, когда репо склонировано к себе) —
ту же работу по умолчанию делает **агент твоего IDE** (Claude Code / Cursor /
Codex): открой репо в IDE, и агент по `AGENTS.md` сам проведёт эксперимент,
заполнит `.run.md` и вольёт его через `--ingest-only`. Плейбуки —
`agents/skills/*/SKILL.md`.

Тот же механизм работает в UI: в панели узла кнопка **«Спросить агента»** — вопрос
уходит в очередь агент-чата, ответ появится в панели.

### 7.2 Путь Б — эксперимент проводишь сам, отчёт пишешь руками

Самый простой вариант — через UI: кликни по карточке канбана, и редактор отчёта
откроется с **преднаполненным шаблоном** — реальные id гипотезы, метода и карточки
и сегодняшняя дата уже подставлены (строка под заголовком редактора подскажет, что
это шаблон). Останется заполнить разделы по образцу ниже и нажать «Сохранить».
Если же пишешь файл руками вне UI — сначала узнай id узлов и карточки. Проще всего — открыть
`projects/<id>/project.md`: id стоят прямо в заголовках узлов и в комментах карточек.
Для эталонного примера (§0):

```markdown
## cause: n-ce100001           ← гипотеза «Перебор по сетке неэффективен»
#### method: n-ce100003        ← метод «Решатель по Виета против перебора»
| … Сравнить скорость и точность… <!-- id:c-ce100010 … --> |   ← карточка (done)
```

Дальше: скопируй шаблон `agents/skills/koi-report-review/experiment-report.md` в
`projects/<id>/reports/<Заголовок узла>/<Заголовок карточки>.run.md`
(пробелы в именах → `_`) и заполни. Строки-подсказки с `>` — удалить.

Для автоинтеграции критичны три места:

1. **§0 «Привязка»** — в бэктиках реальные id: гипотезы (`n-ce100001`), метода
   (`n-ce100003`) и карточки (`c-ce100010`).
2. **§5.1** — строка вердикта строго вида `` `<id-причины>` → … **supported** ``
   (или `refuted` / `open`). Вердикт вешается на узел типа «Причина» (cause) — именно
   по причинам считается прогресс в дашборде базы знаний. Указать id другого узла
   можно, но интеграция предупредит, и в чипы прогресса он не попадёт.
3. **§5.2** — **ровно один** fenced ```json блок с массивом **не более 3** инсайтов.

Ниже — иллюстративный отчёт для такого учебного эксперимента. Подставь собственную
команду и фактически полученные числа.

````markdown
# Отчёт: Виета против перебора по сетке на 1000 случайных уравнений

## 0. Привязка

| Поле | Значение |
|------|----------|
| Гипотеза (cause) | `n-ce100001` — перебор по сетке фундаментально неэффективен |
| Метод / карточка | `n-ce100003` / `c-ce100010` |
| Спека гипотезы | правило решения зафиксировано до прогона (см. §3) |
| Дата прогона | 2026-06-12 |
| Агент-исполнитель | Claude (эталонный пример) |
| Статус прогона | завершён успешно |

## 1. Что запущено (воспроизводимость)

- Команда: `<команда вашего учебного эксперимента>`
- 1000 уравнений ax²+bx+c=0: a∈[−10,10]\{0}, b,c∈[−100,100], seed 42; вещественные корни у 864
- Аналитика: дискриминант + формула корней, самопроверка по Виета (x1+x2=−b/a, x1·x2=c/a)
- Перебор: сетка x∈[−200,200], шаг 1e−2 (40 001 кандидат на уравнение), корень = середина интервала со сменой знака f(x)
- Железо: один поток, чистый Python (без numpy)

## 2. Основная метрика и результаты

| вариант | время на 1000 ур. | точность | найдено корней |
|---------|-------------------|----------|----------------|
| аналитика (Виета) | 0.000164 с | max residual 7.7e−11; Виета-самопроверка 0 расхождений | 100% (1728/1728) |
| перебор по сетке | 2.015 с | ошибка корня до 5.0e−3 (полшага сетки) | 96.9% уравнений целиком; 27 корней (3.1% ур.) вне сетки потеряны |

Сводка одной фразой: аналитика быстрее перебора в ×12 271 и точнее на ~8 порядков;
внутри своей сетки перебор нашёл всё (1701/1701), но 27 корней за границами
[−200,200] потерял молча.

## 3. Проверка правила решения

- Правило (зафиксировано до прогона): **supported**, если аналитика ≥×100 быстрее И residual <1e−9 на ≥99% уравнений; **refuted**, если перебор ≤×10 медленнее при сопоставимой точности; иначе **open**.
- Факт: ускорение ×12 271 (порог ×100); residual <1e−9 на 100% уравнений (порог 99%).
- → правило даёт: **supported**

## 4. Угрозы и оговорки (что этот прогон НЕ доказывает)

- Перебор наивный: адаптивная сетка/бисекция сократили бы разрыв по времени, но не до порога ×10 — и не решили бы проблему корней вне диапазона.
- Шаг 1e−2 — компромисс: мельче сетка → точнее, но ещё медленнее; на вердикт не влияет, разрыв и так на 4 порядка.
- Катастрофическое сокращение при b²≫4ac здесь не ловится — вынесено в отдельную гипотезу (`n-ce100020`).
````

## 8. Шаг 5 — интеграция отчёта в базу знаний

Готовый отчёт вливается одной командой (сначала можно `--dry-run`):

```bash
python -m koi.projects.report_ingest.cli пример-решение-квадратных c-ce100010 --ingest-only
```

Без аргумента `--ingest-only` берёт отчёт по ожидаемому пути
(`reports/<узел>/<карточка>.run.md`); если `.run.md` нет, но есть сохранённый из UI
публичный `.md` — возьмётся он. Можно и явно: `--ingest-only путь/к/отчёту.run.md`.

Что произойдёт автоматически (один проход, потом штатный хук `save_project`):

1. Вердикт **supported** встанет на гипотезу `n-ce100001` → на карте появится бейдж ✔.
2. Два инсайта лягут в `research.json` и появятся в панели метода
   («Выводы по экспериментам»: «Точный ответ» / «Неточный / предварительный»).
3. Карточка `c-ce100010` переедет в **done**, `reports/index.json` получит запись.
4. Пересоберутся `KNOWLEDGE.md`, `knowledge/hypotheses.md` и журнал `KNOWLEDGE_LOG.md`.

Свойства, на которые можно опираться:

- **Идемпотентность**: повторный ingest того же отчёта — точный no-op. Поправил
  отчёт → прогнал снова → инсайты *этой* карточки заменились, чужие не тронуты.
- **Отбраковка**: если нет привязки в §0, строки вердикта в §5.1 или json в §5.2
  кривой (не один блок, >3 инсайтов, синтаксис) — интеграция падает с понятной
  ошибкой «что чинить» и **ничего не меняет**.

## 9. Шаг 6 — где смотреть накопленные знания

- **В UI**: кнопка **«База знаний»** в тулбаре → оглавление проекта (чипы вердиктов,
  прогресс supported/refuted/open по причинам, карточки гипотез с инсайтами,
  документы) и вкладка **«Журнал пополнений»**. Бейджи ✔/✗ видны прямо на карте.
- **В файлах**: `projects/<id>/KNOWLEDGE.md` (оглавление со сводками),
  `knowledge/hypotheses.md` (автоген: все вердикты и инсайты со ссылками на отчёты),
  `KNOWLEDGE_LOG.md` (журнал).
- **Свои документы знаний**: положи любой `.md` в `projects/<id>/knowledge/` —
  подхватится автоматически. Конвенция: `# Заголовок`, первый абзац после него =
  сводка для оглавления. Автоген-файлы (`KNOWLEDGE.md`, `hypotheses.md`,
  `KNOWLEDGE_LOG.md`) руками не править — перезапишутся.
- **По API**: `GET /projects/{id}/knowledge/summary` (это же ест дашборд).

## 10. Публичный отчёт (когда результат стоит показать наружу)

`.run.md` — рабочий слой с локальными путями и сырыми метриками. Если результат
важный, из него делается публичный `<карточка>.md` рядом — по
`agents/skills/koi-report-review/report-skeleton.md` и правилам `report-rules.md`
(без «AI-markdown», публичные имена вместо локальных путей). В UI отчёт карточки
открывается кликом по карточке канбана; картинки и видео из `assets/` рендерятся
прямо в отчёте. Решение, в каком виде инсайт идёт в базу — по матрице `docs/research-workflow.md`.

**Где смотреть основание вердикта/инсайтов.** Если публичного `.md` ещё нет, а
агент уже отработал — клик по карточке покажет его рабочий `.run.md` (строка под
заголовком укажет путь, например `reports/Решатель_по_Виета_против_перебора/….run.md`). Это и есть
первоисточник того, что попало в базу знаний: правило решения, числа, вердикт и
json-инсайты. Сохранение из этого вида создаст публичную версию рядом. Что и когда
было добавлено в базу — в `KNOWLEDGE_LOG.md` (вкладка «Журнал пополнений» в UI).

## 11. Чек-лист и типичные ошибки

Перед запуском эксперимента:
- [ ] Гипотеза — проверяемое утверждение, а не тема.
- [ ] **Правило решения записано ДО прогона** (supported если…; refuted если…; open иначе).
- [ ] Карточка в канбане заведена, id узлов/карточки известны.

Перед интеграцией отчёта:
- [ ] §0: id в бэктиках — реальные (из `project.md`).
- [ ] §3: вердикт следует из подстановки чисел в правило.
- [ ] §5.1: строка вида `` `n-…` → … **supported** ``, id — узла-причины.
- [ ] §5.2: один ```json блок, ≤3 инсайтов, в каждом `method_id`+`card_id`+числа в `answer`.
- [ ] `certainty: definite` только если порог пройден с запасом и вывод устойчив к §4.

Типичные грабли:
- Вердикт «по впечатлению» без правила решения — ревьюер завернёт, метрика обязана
  упираться в порог из спеки.
- Несколько ```json блоков в §5.2 или >3 инсайтов — отчёт отбраковывается целиком.
- Инсайт без чисел в `answer` — бесполезен для следующих экспериментов.
- Ручная правка `KNOWLEDGE.md` / `hypotheses.md` — перезапишется при следующем
  `save_project`; курируемое — только в `knowledge/<свой-файл>.md`.
